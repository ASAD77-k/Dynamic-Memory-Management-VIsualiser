Why does the OS manage memory?
Simple explanation first!
🏠
Imagine your RAM is a flat with 16 rooms. Many apps (Chrome, VS Code, games) all need rooms at the same time. The OS is like a building manager — it decides which app gets which room, moves things around when needed, and makes sure apps don't enter each other's rooms.
1

2
RAM is limited — we can't give every program unlimited space, so the OS must share it fairly.
3
The OS uses clever techniques — Paging, Segmentation, Virtual Memory — to do this efficiently.

📄 Paging — How it Works

Memory is divided into fixed-size chunks called pages. The OS keeps a table to track where each page lives in RAM.

💡
Simple Analogy First
📚
Think of a library with numbered shelves. Your book chapters (pages) can be on ANY shelf — shelf 3, shelf 10, shelf 15. The library keeps a catalogue (page table) that says "Chapter 1 is on shelf 3, Chapter 2 is on shelf 10..." When you want Chapter 2, you check the catalogue → go to shelf 10. That's paging!
1
Your program has a Virtual Address — e.g., address 8200
2
OS splits it: Page Number = 8200 ÷ 4096 = 2  |  Offset = 8200 mod 4096 = 8
3
OS checks the TLB (super fast cache). If page 2 is there → TLB HIT! ⚡ Done instantly.
4
If not in TLB → check Page Table. If found → TLB MISS but OK, just slightly slower.
5
If page not in RAM at all → PAGE FAULT! 🚨 OS loads it from disk. Slow, but OK.
6
Physical Address = Frame Number × 4096 + Offset. Done! Program gets its data.

🗂️ Segmentation

Programs are split into logical parts (segments) of different sizes — code, data, stack. Each segment is stored together in RAM.

💡
Simple Analogy
🏡
Think of a house with different rooms: kitchen, bedroom, living room. Each room is a different size. Code segment = bedroom (where you store your work), Data segment = kitchen (where data lives), Stack = living room (temporary things). The OS knows the address and size of each room.
1
Address = (Segment ID, Offset) — e.g., Segment 0, Offset 500
2
OS looks up Segment Table → gets Base address and Limit (size)
3
Check: Is Offset < Limit? If YES → Physical Address = Base + Offset ✅
4
If Offset >= Limit → SEGMENTATION FAULT! 🚨 Access denied — you went outside the segment boundary.

💿 Virtual Memory

Give programs the illusion of unlimited RAM! Pages not in use are saved to disk and loaded back when needed.

💡
Simple Analogy
📦
Your desk (RAM) is small — only 16 spots. But you have 32 things to work on. Solution: keep the most-used things on your desk, and store the rest in a drawer (disk/swap space). When you need something from the drawer, swap it with something on the desk. That's virtual memory!
1
Program thinks it has 32 pages of memory, even though only 16 physical frames exist.
2
Pages currently being used are In Memory (green). Others wait on disk (orange).
3
When a program accesses a page on disk → Page Fault → OS loads it into a free frame.
4
If RAM is full → OS must evict a page (send it to disk) to make space. Choose wisely!

🔄 Page Replacement

When RAM is full and a new page is needed, which old page do we kick out? Different algorithms give different answers!

💡
The Problem
🎒
Your school bag (RAM) fits only 3 books. You need a 4th book — which one do you take out? FIFO = take out the oldest book. LRU = take out the one you haven't used in the longest time. Optimal = take out the one you won't need the longest (needs future knowledge!).
📅 FIFO
First In, First Out
Remove the page that has been in RAM the longest. Like a queue — first in, first out. Simple but not always smart.
⏰ LRU
Least Recently Used
Remove the page that hasn't been used for the longest time. Smarter — recent usage predicts future usage.
🔮 Optimal
Best Possible
Remove the page that won't be needed for the longest future time. Perfect but impossible in real OS (needs future knowledge!).
